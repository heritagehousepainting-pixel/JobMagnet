# SQLite → Render Postgres Migration — Design

Date: 2026-06-23
Status: Approved (design); pending implementation plan
Scope: Track A of the durability/scale/activation initiative. This single migration
delivers **durability (#1)** and **scale (#2)**. First-win activation (#3) is a
separate sub-project (Track B) with its own spec.

## Problem

JobMagnet runs on a single SQLite file on one Render instance, one gunicorn worker
(single writer), on a 1GB disk with **no backup/disaster-recovery story**. The
SQLite file holds all tenant data *and* the append-only consent ledger (a legal
record for the TCPA/DNC compliance moat). Two consequences:

- **Durability risk:** losing that one file loses everything, including the consent
  ledger. There is no automated backup.
- **Scale ceiling:** single-writer SQLite + one worker cannot credibly back the
  multi-location / Scale tier the product wants to sell.

Moving to **Render Postgres** solves both at once: managed automated backups + PITR
(durability), and real concurrent writers so we can drop the one-worker constraint
(scale).

## Current data reality

The local `jobmagnet.db` is disposable dev/seed data: 1 business (Heritage House
Painting), 1 user, a handful of demo posts/leads/messages, **0 consent_events**, 0
conversions, 0 reviews. No `consent_ledger` table exists yet. This is pre-revenue,
single-owner. **Therefore: no complex data-migration/cutover.** Schema + queries are
ported to Postgres and the database is recreated fresh. (If the deployed Render
instance holds anything, it is tiny and re-seedable / one-time dumpable.)

## Decisions (locked during brainstorming)

1. **Move to Render Postgres now** — collapses durability + scale into one track.
2. **No data preservation** — disposable dev data; fresh recreate.
3. **Clean cutover, single Postgres backend everywhere** — raw `psycopg` v3, no ORM.
   Tests run against an ephemeral Postgres. Rationale: dev == prod, no dialect
   drift, matches the product's "test what you ship / honest states" philosophy. A
   dual-backend (SQLite-dev / Postgres-prod) abstraction was rejected as leaky.

## Architecture

### Connection seam — `db_core.py`
- Add `psycopg[binary]` (v3) and `psycopg_pool` to `requirements.txt`.
- `db_core.py` owns a `ConnectionPool` opened from `DATABASE_URL`, handing out
  connections with `row_factory=dict_row` so existing `row["col"]` access across the
  codebase keeps working **unchanged**. This localizes the backend swap to one file.
- Keep the existing injected-connection-factory pattern (the shared kernel already
  injects `get_conn` into the consent/messaging helpers); it just returns psycopg
  connections now.

### Config — `config.py`
- Replace `JOBMAGNET_DB_PATH` / `DB_PATH` with:
  - `DATABASE_URL` — provided by Render Postgres.
  - `TEST_DATABASE_URL` — ephemeral Postgres for the test suite.

### Dialect rewrite
Across `db.py` and the raw SQL in `consent.py`, `tc_messaging.py` (ledger),
`roi.py`, `reviewsync.py`:
- `?` → `%s` placeholders
- `INSERT OR IGNORE` → `INSERT … ON CONFLICT DO NOTHING`
- `INSERT OR REPLACE` → `INSERT … ON CONFLICT (…) DO UPDATE`
- `INTEGER PRIMARY KEY AUTOINCREMENT` → `BIGINT GENERATED BY DEFAULT AS IDENTITY`
- `cursor.lastrowid` → `INSERT … RETURNING id`
- SQLite datetime (`datetime('now')`, `strftime(...)`) → Postgres (`now()`,
  `to_char(...)`, interval math)
- Remove `PRAGMA` statements; normalize boolean handling (0/1 → `boolean` where
  appropriate, or keep integer columns consistently)

### Schema
Keep the existing `ensure_*` / on-demand DDL pattern, ported to Postgres dialect.
Minimal structural change — same tables, same columns, Postgres types.

### Deployment — `render.yaml`
- Add a Render Postgres `databases:` entry; wire `DATABASE_URL` into the web service.
- Remove the persistent `disk:` and `JOBMAGNET_DB_PATH`.
- Raise gunicorn `--workers` above 1 (Postgres handles concurrent writers).
- Document the restore drill (Render automated backups + PITR).

## Testing — the audit gate
- Tests run against an **ephemeral Postgres** via `TEST_DATABASE_URL`: create a
  fresh schema at suite start, drop at end. Keep the framework-free, exit-0 style.
- **Pass criterion: all 453 checks green on Postgres** (406 in `test_smoke.py` + 47
  in `test_compliance_core.py`).
- Plus a manual smoke: login → create business → send a (simulated) message →
  compute ROI / cost-per-booked-job.

## Components & boundaries
- `db_core.py` — connection pool + factory (the only file that knows it's Postgres).
- `db.py` — data helpers, Postgres-dialect SQL, same public function API as today.
- Kernel modules (`consent.py`, `tc_messaging.py`, `roi.py`, `reviewsync.py`) —
  raw SQL ported; no API changes.
- `config.py` — env wiring.
- `render.yaml` — managed Postgres + relaxed worker constraint.
- Test harness — ephemeral-Postgres setup/teardown.

The public function API of `db.py` does **not** change, so downstream callers
(`app.py`, `assistant.py`, etc.) are largely untouched — this is what keeps Track B
(first-win) buildable on top afterward.

## Out of scope
- First-win activation (#3) — separate spec/plan (Track B).
- Any data migration/ETL (no real data to preserve).
- ORM adoption, schema redesign, or unrelated refactoring.

## Execution / agent plan
Two sequential phases — both touch `db.py`, so they cannot run simultaneously:
1. **Phase 1:** one Sonnet build agent performs this migration in an isolated
   worktree → audit (453 tests on Postgres + manual smoke) → merge.
2. **Phase 2:** short first-win brainstorm → plan → Sonnet build agent on Postgres →
   audit → merge.

## Acceptance criteria
- App runs against Render Postgres with `DATABASE_URL`; no SQLite at runtime.
- All 453 existing checks pass on Postgres.
- `render.yaml` provisions managed Postgres, no persistent disk, workers > 1.
- Automated backups + PITR confirmed available; restore drill documented.
- Consent ledger and all tables exist with correct Postgres types.
- `db.py` public function API unchanged (downstream callers untouched).
